Turn Claude Code into an IDE, in ~30 seconds.

By default Claude Code uses grep for code navigation — fine for strings, wasteful for symbols. Wire up the LSP tool and it does goToDefinition, findReferences, and workspaceSymbol like a real editor. Less token churn, fewer wrong matches, faster work.

One-time setup

# 1. Make the LSP tool env flag permanent
echo 'export ENABLE_LSP_TOOL=1' >> ~/.zshrc   # or ~/.bashrc

# 2. Install a TypeScript language server (pick one)
npm i -g @vtsls/language-server typescript

# 3. Register the official Claude LSP marketplace
claude plugin marketplace add Piebald-AI/claude-code-lsps

# 4. Install + enable the TS LSP plugin at user scope
claude plugin install typescript-lsp@claude-plugins-official --scope user
claude plugin enable  typescript-lsp@claude-plugins-official --scope user

# 5. Reload shell
source ~/.zshrc
Other languages: same pattern. Python: pip install pyright + pyright-lsp plugin. Go: go install golang.org/x/tools/gopls@latest + gopls-lsp. Rust / Kotlin plugins also live in that marketplace.

Verify

claude plugin list | grep -i lsp
echo "ENABLE_LSP_TOOL=$ENABLE_LSP_TOOL"
command -v vtsls

All three should return something. Restart Claude Code after the setup so the plugin + env picks up.

Make the agent actually use it

Drop this line into your CLAUDE.md (global or per-project):

## Code navigation
Prefer LSP tools (goToDefinition, findReferences, workspaceSymbol)
over Grep for symbol navigation. Fall back to Grep only for
strings, comments, or non-code artifacts.

Why bother

  • Exact semantic results instead of string matches that hit shadowed names or comments.
  • Drops token cost dramatically — one definition lookup replaces 3–5 grep passes across src/ and node_modules.
  • Works across package boundaries — follow a call from your code into a dependency's types without guessing paths.

Tested on Claude Code with vtsls 0.3.0. Free. No backend. No telemetry to us.
